# CLAUDE.md — sonar-l10n-zh-tw

## Project Overview

`sonar-l10n-zh-tw` is the **Traditional Chinese (zh-TW) Language Pack** for SonarQube. It ships as a `sonar-plugin` JAR containing:

1. A minimal Java plugin entry point (`TraditionalChinesePackPlugin`) that registers the bundle with SonarQube.
2. Localized `.properties` resource bundles consumed by SonarQube's i18n framework.

Most ongoing work is **translation** (keeping `core_zh_Hant.properties` aligned with upstream `core.properties`), not Java code changes.

- **Language:** Java 21 (build target)
- **Build tool:** Maven 3.x
- **Plugin API:** `sonar-plugin-api` 9.8.0.203
- **Target SonarQube:** 25.3+ / 26.x
- **Packaging:** `sonar-plugin` (produces `sonar-l10n-zh-tw-plugin-<version>.jar`)
- **License:** GNU LGPL 3

## Development Commands

```bash
# Compile + run tests (includes I18nMatchers.assertBundlesUpToDate())
mvn clean test

# Build the plugin JAR
mvn clean package

# Verify with coverage (jacoco)
mvn clean verify

# Skip tests (only when explicitly justified)
mvn clean package -DskipTests
```

The `native2ascii-maven-plugin` runs during `process-resources` and converts UTF-8 source `.properties` files to ASCII-escaped form in `target/classes/`. Edit the **UTF-8 sources** under `src/main/resources/`, never the generated files under `target/`.

## Architecture Notes

This plugin is intentionally minimal — it does **not** add any extensions:

- `TraditionalChinesePackPlugin.define()` calls `context.addExtensions(Collections.emptyList())`.
- SonarQube's i18n loader discovers `core_zh_Hant.properties` automatically based on filename locale suffix (`_zh_Hant`).
- The plugin class exists only so Maven's `sonar-packaging-maven-plugin` produces a valid SonarQube plugin JAR (configured via `sonar.pluginClass`).

**Do not add extension points, services, or scanner sensors** to this plugin. If a future requirement needs them, raise it explicitly — adding them silently changes the plugin's role from "language pack" to "functional plugin," which has different review and distribution implications.

## Translation Standards (PRIMARY WORK)

### Locale & Encoding

- Target locale: **`zh_Hant`** (Traditional Chinese, Hant script). Filename suffix MUST be `_zh_Hant`, not `_zh_TW` or `_zh`.
- Source files use **UTF-8** encoding. The `native2ascii-maven-plugin` handles the conversion to ASCII-escaped form at build time.
- Line endings: respect the existing file (LF). Do not introduce mixed CRLF/LF.

### Translation Quality

- **Use Traditional Chinese characters only.** Never Simplified. (e.g., `設定` not `设置`; `軟體` not `软件`; `資料` not `数据`.)
- **Use Taiwan Mandarin terminology**, not mainland conventions. Common substitutions:
  - 軟體 (not 軟件 / 软件) — software
  - 資訊 (not 信息) — information
  - 程式 (not 程序) — program
  - 滑鼠 / 螢幕 / 網路 (not 鼠标 / 屏幕 / 网络)
  - 預設 (not 默認) — default
  - 套件 (not 插件 / 包) — plugin / package
- **Preserve placeholders verbatim.** Tokens like `{0}`, `{1}`, `%s`, `%d`, HTML tags (`<a>`, `<b>`), and Markdown markers must remain unchanged in count and order.
- **Preserve product names and CLI tokens in their original form** — `SonarQube`, `SonarLint`, `Bitbucket`, `GitHub`, `OAuth`, `LDAP`, `JWT`, `Quality Gate` (or use the established convention from existing entries).
- **Preserve trailing/leading whitespace** in property values — it is often significant for UI rendering.
- **Match the key set of `core.properties` exactly.** No extra keys, no missing keys (the bundle freshness test will fail otherwise).

### Consistency

- Reuse translations already established in `core_zh_Hant.properties`. Search the file for an existing entry before inventing new wording for the same English term.
- When upstream adds new keys, add them to `core.properties` (English baseline) AND `core_zh_Hant.properties` in the same commit. Keep them in approximately the same order/section as upstream.

## Version & Release Workflow

The plugin version tracks the SonarQube version it targets (e.g., `26.4` targets SonarQube 26.4). When bumping:

1. Update `<version>` in `pom.xml`.
2. Update `<tag>sonar-l10n-zh-tw-X.Y</tag>` in `pom.xml` `<scm>` block to match.
3. Update the **Compatibility Matrix** in `README.md` — add a new column for the new SonarQube version mapped to the new plugin version.
4. Sync any new/changed keys from upstream SonarQube's `core.properties` into both `core.properties` (English) and `core_zh_Hant.properties` (zh-Hant translation).
5. Run `mvn clean test` and verify `bundles_should_be_up_to_date` passes.
6. Run `mvn clean package` to produce the release JAR.
7. Commit with message `Update version X.Y`.

## Coding Standards (Java side)

The Java surface is tiny, but any change to it must follow these rules.

### Software Engineering Principles

- **SOLID:** The plugin class is a single-purpose adapter to SonarQube's `Plugin` interface — keep it that way.
- **DRY:** No duplication; the entire Java footprint should remain ≤ 100 lines.
- **KISS:** Prefer the existing minimal pattern over abstractions. Resist adding helper classes.
- **YAGNI:** Do not add extension points, configuration, or services unless explicitly requested.

### Static Analysis Compliance (SonarQube Java rules)

This is a SonarQube plugin — it would be embarrassing to fail SonarQube's own analysis. Java code MUST pass without new issues.

#### Complexity & Maintainability

- **Cognitive complexity ≤ 15** per method (`java:S3776`)
- **Cyclomatic complexity ≤ 10** per method
- **Method length ≤ 75 lines**; **class length ≤ 750 lines** (`java:S138`)
- **Method parameters ≤ 7** (`java:S107`)
- **Max nesting depth = 4** (`java:S134`); use early returns / guard clauses
- **No duplicated code blocks ≥ 3 lines** (`java:S4144`); extract helpers

#### Naming & Style (Java conventions)

- Classes: `PascalCase`; methods / fields / locals: `camelCase`; constants: `UPPER_SNAKE_CASE` (`java:S100`, `java:S101`, `java:S116`)
- No single-letter names except loop indices `i`, `j`, `k`
- Line length ≤ 120 chars; 4-space indentation; no tabs
- No wildcard imports (`import a.b.*`) — `java:S2208`
- One statement per line

#### Bug Prevention

- **Always close resources** — use try-with-resources for `AutoCloseable` (`java:S2095`)
- **Never catch `Exception` or `Throwable` broadly** without re-throwing or logging context (`java:S2221`, `java:S1181`)
- **Never silently swallow exceptions** — empty `catch` blocks are forbidden unless the exception type is a documented expected control-flow signal (`java:S108`)
- **No `==` comparison on `String` or boxed primitives** — use `.equals()` (`java:S4973`)
- **No identical expressions on both sides** of `==`, `!=`, `&&`, `||` (`java:S1764`)
- **No unreachable code** after `return` / `throw` / `break` / `continue` (`java:S1763`)
- **No raw types** — always parameterize generics (`java:S3740`)
- **No assignment in conditions** (`if (x = y)`) — `java:S1121`
- **`equals` and `hashCode` must be overridden together** (`java:S1206`)
- **Override annotations required** for overridden methods (`java:S1161`)

#### Security

- **Never use `Runtime.exec` or `ProcessBuilder` with user input** (`java:S4036`, `java:S2076`)
- **No insecure hash algorithms** (MD5, SHA-1) for security purposes — use SHA-256+ (`java:S4790`)
- **No hardcoded credentials, tokens, or URLs with secrets** (`java:S2068`, `java:S1313`)
- **No insecure TLS** — never disable certificate validation (`java:S4830`)
- **No predictable random** for security purposes — use `SecureRandom`, not `Random` (`java:S2245`)
- **XML parsing must disable external entities** (XXE prevention) — `java:S2755`
- **Never use `eval`-like dynamic class loading** on untrusted input

#### Documentation

- Public classes and non-trivial public methods require Javadoc
- No `// TODO` / `// FIXME` without an associated issue link (`java:S1135`)

#### Testing Quality

- Test methods must contain at least one assertion (`java:S2699`)
- Never use `assertTrue(true)` as a placeholder (`java:S2699`)
- Disabled tests (`@Ignore`, `@Disabled`) require a documented reason

### Code Quality (general)

- Remove dead code, unused imports, and commented-out blocks — do not leave them in the codebase
- Keep the plugin class focused; extract new classes only when a clear boundary exists
- Final fields and immutability by default
- Use `final` on the plugin class (already the case — keep it)

### Pre-Commit Verification

Before committing, run the full Maven build and ensure it passes cleanly:

```bash
mvn clean verify
```

This runs compilation, unit tests (including `I18nMatchers.assertBundlesUpToDate()` which validates the bundle), and JaCoCo coverage. If SonarCloud analysis fails on a PR, fix the new issues in the same PR — do not defer them with `@SuppressWarnings` unless a code comment explains the justification.

## Git & Commit Conventions

- Branch model: `main` (stable) — feature branches like `l10n_new` for translation updates merge in via PR
- Upstream remote: `SDPM-lab/sonar-l10n-zh-tw`; fork remote: `JE-Chen/sonar-l10n-zh-tw`
- Commit messages: concise, imperative mood
  - Version bumps: `Update version X.Y` (matches existing repo history)
  - Translation work: `Add translations for <area>`, `Fix <key> translation`
  - Code changes: `Add element validation`, `Fix driver cleanup on timeout` style
- **Do NOT mention any AI tools, assistants, or language models in commit messages** — commits must read as standard developer-authored messages
- **Do NOT include `Co-Authored-By` lines referencing AI in commits**
- **Track which build artifacts to include carefully.** Historically this repo has committed parts of `target/` (e.g., classes, test reports, jacoco.exec) alongside source changes during version bumps. Continue that pattern only for files that already have git history; do **not** start tracking new build artifacts that have never been committed before.

## Testing

- Tests are located in `src/test/java/org/sonar/plugins/l10n/`
- Test framework: **JUnit 4** (`junit:junit:4.13.2`)
- The critical test is `bundles_should_be_up_to_date` — it uses SonarQube's `I18nMatchers.assertBundlesUpToDate()` to verify every key in `core.properties` has a corresponding translation in `core_zh_Hant.properties` (and vice versa)
- The bundle freshness check writes a diff report to `build/l10n/core_zh_Hant.properties.report.txt` — review this file when the test fails to see exactly which keys are missing or stale
- A second smoke test (`testTraditionalChinesePackPlugin`) exercises the plugin's `define()` path purely for coverage

When adding a new translation key, run `mvn test` locally and confirm the bundle test passes before pushing.
